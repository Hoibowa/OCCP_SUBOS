# ホワイトペーパー：オンチェーン・コンテキスト・プロトコル（OCCP）
**〜「価値の移転」と「情報の保存」を等価とする新しい情報の物理学〜**

## 1. 概念提唱（The Concept）

### 1.1 背景：中央集権インフラの限界と「維持費」という重力
現代のデジタルインフラにおいて、情報は常に「サーバー維持費」という継続的な固定コストの重力に縛られている。企業や個人がそのコスト負担を停止した瞬間、いかに価値のある記録であってもネットワーク上から完全に消滅する。非中央集権を謳うWeb3プロジェクトの多くも、実際にはフロントエンドやインデックス用データベースを特定のサーバーに依存しており、真の意味での「自律的な永続性」は達成されていない。

### 1.2 解決策：「流動性」によるインフラの自己金融化
本プロトコル（OCCP: On-Chain Context Protocol）は、「通貨の流動性（トランザクション）」そのものを、情報の記録・保存の動力源に変えるパラダイムシフトを提唱する。ネットワーク上で価値が移転する際の「トランザクションの手数料（ネットワークコスト）」を、ストレージの維持費として直接機能させる。これにより、経済圏（ネットワーク）が稼働している限り、誰の許可も継続的な支払いも必要とせずに情報が永続化される「自律循環型インフラ」が構築される。

## 2. コア・アーキテクチャ（The Architecture）

本プロトコルは、特定のブロックチェーンの仕様に依存しない普遍的な分散型データ構造として、以下の設計原則（アーキテクチャ）を定義する。

### 2.1 レジャー・マージンへのデータ・カプセル化
あらゆる分散型台帳（パブリックチェーン、企業間コンソーシアム、通貨システム）が備える「トランザクションの任意データ領域（メタデータやメモ欄）」を、情報の保存セルとして利用する。送金という経済活動の中に暗号化されたテキストをカプセル化し、ネットワークの合意形成プロセスに相乗りする形で半永久的なデータ保存を実現する。

### 2.2 ネイティブ・クエリによるインデックス最適化と文脈生成
パブリック台帳のデータは膨大であり、特定の文脈（ストリーム）をクライアント側で全件検索することは非現実的である。真の分散型インフラとして機能するためには、**ノードが標準で提供するAPI（RPCコール）のみを活用し、クライアント側の演算・フィルタリング処理を最小化する設計**が不可欠となる。

本アーキテクチャは、以下の構造によってこれを解決する。
*   **コンテキスト・リンク（Linked List）**: 個別のトランザクション内に「直前の記録のハッシュ値（親ポインタ）」を内包させる。これにより、一つの起点を見つければ、過去に向かって数珠つなぎにデータを取得するだけで文脈（ダイアリーやブログ等）が論理的に復元される。
*   **状態ポインタによるHEAD管理**: ノードの標準クエリで最も低負荷に取得できるリソース（保有トークンやアカウントステートなど）を利用し、対象ストリームの「最新のトランザクションハッシュ（HEAD）」を記録・更新する。これにより、外部の検索サーバーに依存することなく、ノードへの最小のクエリ発行で即座に情報の最新位置を特定できる。

## 3. 実証アーキテクチャ：XRPLにおけるハックと実装（SUB OS）

本章は前章で提示したOCCPを実験的に私が実装した際の記録としての位置づけである。存在証明の様なものである。

前章で定義した普遍的なアーキテクチャが、既存の金融インフラ上でいかに稼働し得るか。
我々はその極端な実証実験（PoC）として、XRP Ledger（XRPL）の仕様を極限までハックした統合ターミナル「SUB OS」を構築した。
XRPは安価でメモ欄が1kbほどの余力があり、なおかつ社会的にも非常に普及してるためである。

XRPLの標準ノード（rippled）のRPCには、「特定のメモテキストを含むトランザクションを検索する」といった高度なクエリは存在しない。
そのため、我々はこの制約を以下の「回りくどくも合理的なハック」によって突破した。

### 3.1 データ層：メモ欄への射出（The Shouting）
XRPLの超低コストな `Payment` トランザクションを利用し、送金メモ（`MemoData`）にテキストデータをHEX（16進数）変換して書き込む。
この際、プロトコル識別子と「親トランザクションのハッシュ（64文字）」を先頭に付与することで、強固なコンテキスト・リンクを形成している。

### 3.2 インデックス層：NFTを用いた状態ポインタの定着（The Pinning）
XRPLの標準RPCにおいて、最も高速かつ低負荷で情報をフィルタリングできるコマンドの一つが、特定アカウントの保有NFT一覧を取得する `account_nfts` である。
我々はこの仕様を逆手に取り、**本来アートや権利証明に使われるNFTを「ストリームの最新位置を示すインデックス・ポインタ」として再定義**した。
新規投稿のたびに「ラベル名」と「最新のトランザクションハッシュ」をURIに刻んだ新しいNFTを発行（Mint）し、古い同名NFTを破棄（Burn）する。これにより、クライアントはウォレットのNFT一覧を一度クエリするだけで、無数のストリームの最新状態をレンダリングすることが可能となった。

## 4. 展望（The Vision）

本プロトコル（OCCP）の適用範囲は、XRPLという特定のパブリック・チェーンに限定されるものではない。
中央集権的な独自通貨を発行する組織（企業コンソーシアム、ゲーム内経済網、国家のCBDCなど）においても、
この「トランザクションによるインフラ担保モデル」を採用することで、膨大なサーバー運用コストを「経済圏の流動性（トランザクション）の中に溶け込ませる」ことが可能となる。
経済活動が行われるたびに、自動的に情報インフラが補強・維持されていくのである。

中央集権的サーバーと結び付けて応用することも可能だ。
チェーンの方式も、一つのNFTやメモに大量に別のメモ等ハッシュを組み込めば、合理化していくことも可能であろう。



データがサーバーの物理的な檻から解放され、ネットワークを流れる血液（流動性）に乗って永遠に循環し続ける世界。
すなわち、「価値の移転」と「情報の保存」が完全に等価となる、新しい『情報の物理学』の確立である。
